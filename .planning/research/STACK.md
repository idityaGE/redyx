# Stack Research

**Domain:** Anonymous community discussion platform (Reddit clone) — Go microservices, Astro+Svelte frontend
**Researched:** 2026-03-02
**Confidence:** HIGH (all versions verified from official GitHub releases/tags as of Feb/Mar 2026)

## Recommended Stack

### Language & Runtime

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| Go | 1.26.0 | All 12 microservices | Latest stable. Native concurrency, zero-cost gRPC integration, static binaries for containers, exceptional toolchain. Go 1.26 released Feb 2026 — use it from day one. |
| Node.js | 22 LTS | Frontend build tooling (Astro/Svelte) | Required for Astro SSR. Use LTS for stability. |

### gRPC & Protocol Buffers

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| grpc-go | v1.79.1 | `google.golang.org/grpc` | Official Go gRPC. Released Feb 2026. All 12 services communicate over gRPC internally. Supports server reflection (critical for Envoy discovery), interceptors for auth/logging/tracing, and health checking protocol. |
| protoc-gen-go-grpc | v1.6.1 | `google.golang.org/grpc/cmd/protoc-gen-go-grpc` | Code generator for gRPC service stubs. Paired with grpc-go. |
| protoc-gen-go | latest | `google.golang.org/protobuf/cmd/protoc-gen-go` | Go protobuf message code generation. Use `google.golang.org/protobuf` (NOT the deprecated `github.com/golang/protobuf`). |
| buf | v1.66.0 | CLI tool | **Use instead of raw protoc.** Buf handles linting, breaking change detection, code generation, and dependency management for .proto files. Far superior DX to managing protoc + plugins manually. |
| grpc-gateway | v2.28.0 | `github.com/grpc-ecosystem/grpc-gateway/v2` | NOT used directly (Envoy does REST-to-gRPC transcoding), but its protoc-gen-openapiv2 plugin generates OpenAPI specs from proto files — useful for frontend API documentation. |

**Integration: gRPC + Envoy**
- Enable gRPC server reflection on all services (`reflection.Register(s)`) — Envoy uses this for dynamic route discovery.
- Proto files must include `google.api.http` annotations for Envoy's gRPC-JSON transcoder filter to map REST → gRPC.
- Use `grpc-health-probe` for Kubernetes liveness/readiness checks over gRPC health checking protocol.
- Envoy's `envoy.filters.http.grpc_json_transcoder` converts REST/JSON from frontend to gRPC for backend services.

### API Gateway

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| Envoy | v1.37.0 | API gateway, REST-to-gRPC transcoding, rate limiting, TLS termination | Production-grade L7 proxy. Native gRPC-JSON transcoder (no custom gateway code). Built-in rate limiting, circuit breaking, mTLS. Released Jan 2026. Docker image: `envoyproxy/envoy:v1.37.0`. |

**Envoy Configuration Considerations:**
- Use `grpc_json_transcoder` filter with proto descriptor set (generated by `buf build -o descriptor.binpb`).
- Rate limiting: Use Envoy's `envoy.filters.http.local_ratelimit` for per-connection limits AND external rate limit service for distributed limits (the Rate Limiting service can serve this).
- Route configuration maps `/api/v1/auth/*` → auth-service gRPC, `/api/v1/posts/*` → post-service gRPC, etc.
- Enable `envoy.filters.http.cors` for frontend cross-origin requests.
- WebSocket upgrade support for notification-service: configure `upgrade_configs` with type `websocket` on the notification route.

### Database Drivers

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| pgx | v5.8.0 | `github.com/jackc/pgx/v5` | **THE PostgreSQL driver for Go.** Pure Go, fastest performance, native PostgreSQL protocol, built-in connection pooling via `pgxpool`. Used by auth-service, user-service, community-service, post-service (sharded), platform-service. Dec 2025 release. |
| pgxpool | (included in pgx v5) | `github.com/jackc/pgx/v5/pgxpool` | Connection pool for pgx. Configure `MaxConns`, `MinConns`, `MaxConnLifetime`, `MaxConnIdleTime`. Each service gets its own pool to its own PostgreSQL instance. |
| scylladb/gocql | v1.17.1 | `github.com/scylladb/gocql` | **ScyllaDB's fork of gocql** — NOT `github.com/gocql/gocql`. The ScyllaDB fork adds shard-aware connection pooling (connections are pinned to specific CPU shards), tablet support, and driver-level optimizations. Used by comment-service. Jan 2026 release. |
| golang-migrate | v4.19.1 | `github.com/golang-migrate/migrate/v4` | Database migrations for PostgreSQL. Supports up/down migrations, versioning, CLI tool. Nov 2025 release. Used across all 5 PostgreSQL instances. |

**Integration: pgx + Sharding (Post Service)**
- Use `pgxpool` for each shard connection. Post-service maintains a pool per shard.
- Application-level consistent hashing on `community_id` determines shard routing.
- Wrap shard selection in a `ShardRouter` that maps `community_id → shard_index → *pgxpool.Pool`.
- Use `pgx.Batch` for multi-row inserts when fan-out querying across shards.

**Integration: ScyllaDB gocql**
- Configure `gocql.ClusterConfig` with `PoolConfig.HostSelectionPolicy` using `TokenAwareHostPolicy(DCAwareRoundRobinPolicy())`.
- Set `NumConns` per host (default 2 per CPU shard — the driver auto-detects shard count).
- Use `QUORUM` consistency for writes, `LOCAL_ONE` or `LOCAL_QUORUM` for reads.
- Comment tree model: partition key = `(post_id)`, clustering key = `(parent_id, created_at, comment_id)` — enables efficient tree retrieval.

### Caching

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| go-redis | v9.18.0 | `github.com/redis/go-redis/v9` | Official Redis Go client. Supports all Redis commands, pipelining, pub/sub, Lua scripting, cluster mode, connection pooling. Feb 2026 release. Used across 6 logical Redis databases (sessions, vote counts, rate limiting, caching, user feeds, leaderboards). |

**Integration: Redis Across Services**
- Use `redis.NewClient()` per service with `DB` option to select logical database (0-5).
- Vote service: Redis `INCR`/`DECR` for real-time vote counts, Lua scripts for atomic vote-change operations.
- Rate limiting: Token bucket in Redis using Lua scripts (atomic check-and-decrement).
- Session storage: Store JWT refresh token metadata with TTL (7 days).
- Use `redis.Pipeline()` for batching multiple operations (e.g., fetching vote counts for a page of posts).

### Message Queue

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| franz-go | v1.20.7 | `github.com/twmb/franz-go` | **Pure Go Kafka client.** No CGO dependency (unlike confluent-kafka-go which requires librdkafka). Faster, lighter, better error handling, feature-complete. Supports consumer groups, transactions, exactly-once semantics. Feb 2026 release. |
| franz-go kadm | v1.17.2 | `github.com/twmb/franz-go/pkg/kadm` | Admin client for topic management. Used to create/manage Kafka topics programmatically. |

**Why franz-go over alternatives:**
- `confluent-kafka-go`: Requires CGO + librdkafka. Breaks static builds, complicates Docker images, slower compilation. Only choose if you need Confluent Schema Registry integration.
- `segmentio/kafka-go`: Simpler API but fewer features, slower consumer group rebalancing, less active maintenance.
- `franz-go`: Pure Go, zero CGO, full Kafka protocol support, fastest benchmarks, excellent maintainer.

**Integration: Kafka Event Topics**
- Topics: `vote.events`, `karma.updates`, `post.events`, `comment.events`, `notification.events`, `moderation.events`, `search.index`.
- Consumer groups: One per consuming service (e.g., `karma-consumer-group`, `notification-consumer-group`, `search-index-consumer-group`).
- Use `kgo.ConsumeTopics()` with `kgo.ConsumerGroup()` for at-least-once delivery.
- Configure `kgo.SessionTimeout(10 * time.Second)` and `kgo.RebalanceTimeout(15 * time.Second)` for consumer group rebalancing.
- Idempotent producers: Enable with `kgo.RequiredAcks(kgo.AllISRAcks())` + `kgo.RecordRetries(3)`.

### Search

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| meilisearch-go | v0.36.1 | `github.com/meilisearch/meilisearch-go` | Official Meilisearch Go SDK. Type-safe, async indexing, faceted search. Feb 2026 release. Search service indexes posts, comments, communities from Kafka events. |

**Integration: Meilisearch**
- Create indexes: `posts`, `communities`, `comments` with appropriate searchable/filterable attributes.
- Search service consumes `post.events` and `comment.events` Kafka topics to index/deindex content.
- Configure `filterable_attributes` for community-scoped search, `sortable_attributes` for relevance/date sorting.
- Use async task API: `client.Index("posts").AddDocuments()` returns a task ID — poll for completion.

### Authentication & Security

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| golang-jwt | v5.3.1 | `github.com/golang-jwt/jwt/v5` | **The standard JWT library for Go.** Community-maintained successor to dgrijalva/jwt-go. Supports RS256/ES256/HS256, custom claims, validation. Jan 2026 release. |
| argon2 (stdlib) | (Go stdlib) | `golang.org/x/crypto/argon2` | Password hashing. argon2id is the recommended algorithm (memory-hard, resistant to GPU attacks). Part of Go's extended stdlib — no external dependency. |
| crypto/aes (stdlib) | (Go stdlib) | `crypto/aes` + `crypto/cipher` | AES-256-GCM for email encryption at rest. Use Go's stdlib — do NOT use third-party crypto libraries. |
| oauth2 | latest | `golang.org/x/oauth2` | Google OAuth2 flow. Part of Go's extended stdlib. Handles token exchange, refresh. |
| pquerna/otp | v1.4+ | `github.com/pquerna/otp` | TOTP/HOTP for OTP verification (email verification codes). Widely used, stable. |

**Integration: Auth Flow**
- Auth-service issues JWT access tokens (15min, HS256 or RS256) + refresh tokens (7 days, stored in Redis).
- All other services validate JWTs via a shared gRPC interceptor (unary + stream).
- The interceptor extracts user_id and role from JWT claims, populates context metadata.
- Envoy can do initial JWT validation via `envoy.filters.http.jwt_authn` filter, but services MUST re-validate for defense in depth.

### WebSocket

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| coder/websocket | v1.8.14 | `github.com/coder/websocket` | **Formerly nhooyr/websocket**, now maintained by Coder. Minimal, idiomatic Go API. Uses `net/http` for handshake (no custom server needed). Supports compression, contexts, concurrent writes. Sep 2025 release. |

**Why NOT gorilla/websocket:**
- gorilla/websocket is unmaintained (archived). coder/websocket is actively maintained, smaller API surface, better integration with `context.Context` and `net/http`.

**Integration: Notification Service**
- Notification-service upgrades HTTP connections to WebSocket at `/ws/notifications`.
- Envoy routes WebSocket upgrade requests to notification-service (configured via `upgrade_configs`).
- Server pushes notification events (new replies, votes, mod actions) to connected clients.
- Use Redis pub/sub as the fan-out mechanism: notification events published to Redis channels per user, WebSocket connections subscribe to their user's channel.

### Observability

| Technology | Version | Import Path | Why Recommended |
|------------|---------|-------------|-----------------|
| OpenTelemetry Go | v1.40.0 | `go.opentelemetry.io/otel` | Vendor-neutral instrumentation. One SDK for traces, metrics, logs. Exports to Jaeger (traces), Prometheus (metrics), Loki (logs). Feb 2026 release. |
| OpenTelemetry gRPC instrumentation | latest | `go.opentelemetry.io/contrib/instrumentation/google.golang.org/grpc/otelgrpc` | Auto-instruments gRPC calls with spans. Add as a gRPC interceptor. |
| zap | v1.27.1 | `go.uber.org/zap` | Structured logging. Fastest Go logger. JSON output for Loki ingestion. Nov 2024 release (stable, mature). |
| prometheus client_golang | v1.20+ | `github.com/prometheus/client_golang/prometheus` | Expose `/metrics` endpoint per service. Prometheus scrapes metrics. Use for custom business metrics (vote counts, post creation rate, etc.). |

**Integration: Observability Stack**
- Each service initializes OTel tracer + meter providers in `main()`.
- gRPC interceptors (`otelgrpc.UnaryServerInterceptor()`, `otelgrpc.StreamServerInterceptor()`) auto-create spans.
- Trace context propagates via gRPC metadata headers (`traceparent`, `tracestate`).
- Envoy propagates trace headers to backend services.
- Zap logger writes structured JSON; Promtail/Loki collects from stdout.
- Grafana dashboards visualize Prometheus metrics + Loki logs + Jaeger traces.

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|------------------------|
| franz-go (Kafka) | confluent-kafka-go | When you need Confluent Schema Registry integration or enterprise Confluent support. Requires CGO + librdkafka. |
| franz-go (Kafka) | segmentio/kafka-go | Simpler projects with basic produce/consume needs. Less feature-complete, slower rebalancing. |
| pgx (PostgreSQL) | database/sql + lib/pq | If you need stdlib `database/sql` interface compatibility for ORM integration. pgx also implements `database/sql` via `pgx/stdlib` adapter. |
| scylladb/gocql | gocql/gocql (upstream) | If using Cassandra instead of ScyllaDB. The upstream driver lacks shard-aware optimizations. |
| coder/websocket | gorilla/websocket | Never — gorilla/websocket is archived/unmaintained. |
| zap (logging) | zerolog | When you want even lower allocations and a simpler API. zerolog is slightly faster but less feature-rich. Either works. |
| zap (logging) | slog (stdlib) | For simpler projects. Go 1.21+ `log/slog` is in stdlib but lacks the performance and ecosystem of zap. |
| buf (proto) | raw protoc | Never — buf is strictly superior for proto management, linting, and generation. |
| Astro + Svelte | Next.js + React | If the team is React-experienced. Heavier JS bundle, more complex hydration. Astro+Svelte ships less JS for content-heavy sites. |
| ko (containers) | Dockerfile multi-stage | When you need custom system dependencies in the container. ko only works for pure Go binaries. |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| `github.com/golang/protobuf` | Deprecated since 2020. The `google.golang.org/protobuf` module is the successor with Editions support. | `google.golang.org/protobuf` |
| `github.com/dgrijalva/jwt-go` | Unmaintained, known security vulnerabilities. | `github.com/golang-jwt/jwt/v5` |
| `github.com/gorilla/websocket` | Archived, no longer maintained. | `github.com/coder/websocket` |
| `github.com/golang/mock` | Deprecated, moved to `go.uber.org/mock`. | `go.uber.org/mock` |
| `github.com/gocql/gocql` (upstream) | Missing ScyllaDB shard-aware optimizations. Performance penalty of ~30-50% on ScyllaDB. | `github.com/scylladb/gocql` |
| `github.com/go-redis/redis` (old path) | Migrated to `github.com/redis/go-redis`. Old path is a redirect. | `github.com/redis/go-redis/v9` |
| `aws-sdk-go` (v1) | Legacy SDK, v2 is the current generation with better performance, modularity, and context support. | `github.com/aws/aws-sdk-go-v2` |
| `gin`, `echo`, `fiber` (HTTP frameworks) | Unnecessary. Envoy handles REST→gRPC transcoding. Services expose gRPC only. HTTP frameworks add bloat. | Raw gRPC server (`google.golang.org/grpc`) |
| `GORM`, `ent`, `sqlc` (ORMs/query builders) | For PostgreSQL with pgx, write raw SQL. ORMs add overhead, hide query behavior, complicate sharding. pgx's type-safe scanning + prepared statements are sufficient. | `pgx` with raw SQL + `pgx.NamedArgs` |
| `RabbitMQ` (via amqp091-go) | Kafka is chosen for durable event log, replay capability, and multi-consumer fan-out. RabbitMQ is better for task queues but worse for event sourcing. | franz-go (Kafka) |
| `protoc` (raw) | Managing protoc plugins, include paths, and generation scripts is error-prone. | `buf` |
