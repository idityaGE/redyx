<decisions>
## Implementation Decisions

### Content density
- Information-rich, terminal-like aesthetic. Dense data per screen, not spacious cards
- Feed items show title + metadata only (author, community, score, comment count, timestamp). No body preview in feeds
- Media displays as small thumbnails on the left side of the post row (old Reddit style). Text stays dominant
- Vote arrows positioned on the left side of each post/comment (Reddit classic muscle memory)
- Overall density closer to old.reddit.com / Hacker News than new Reddit or Lemmy

### Visual identity
- TailwindCSS for styling with built-in dark/light mode support. Both modes are first-class
- Orange/amber as the primary accent color for interactive elements (links, active states, vote highlights, buttons)
- Full monospace typography everywhere — headings, body, metadata, navigation. Use JetBrains Mono, Fira Code, or similar coding font
- ASCII-style box-drawing borders and dividers (+-| characters or CSS equivalents). Post separators, card borders, and section dividers should evoke terminal output
- The overall feel is "a TUI app that happens to live in the browser" — developer/power-user focused, minimal decorative chrome

### Layout & navigation
- Sidebar + top bar pattern (Reddit desktop). Left sidebar with joined communities list and quick shortcuts (Home, Popular, Saved, Profile, Settings). Top bar with search, notifications, and user menu
- SPA-style panel navigation: sidebar stays fixed, content area swaps without full page reloads. Faster navigation, feels like an IDE/terminal with panes
- On mobile: bottom tab bar (Home, Communities, Search, Notifications, Profile) instead of hamburger menu. More app-like, easier thumb reach
- Sidebar collapses on tablet, bottom tabs on mobile phone widths

### API naming conventions
- Flat resource paths: /api/v1/posts, /api/v1/communities, /api/v1/users, /api/v1/comments. Query params for filtering (e.g., /api/v1/posts?community_id=abc)
- gRPC status codes mapped to HTTP errors. Envoy transcoding handles the mapping. Error responses include code, message, and details array per gRPC status convention
- Cursor-based pagination for all list endpoints: ?cursor=abc123&limit=25. No offset-based pagination. Stable for feeds with constantly changing content
- API versioning via URL path (/api/v1/...)

### Claude's Discretion
- JSON field naming convention (snake_case vs camelCase) — pick based on Envoy transcoding defaults and proto conventions
- Loading skeleton/spinner design within the terminal aesthetic
- Error state UI patterns
- Exact spacing and padding values within the monospace grid
- Sidebar collapse/expand animation style
- Specific monospace font choice (JetBrains Mono vs Fira Code vs others)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- None (greenfield project — no existing code)

### Established Patterns
- Research recommends monorepo layout: single go.mod, proto/ at root, gen/ for generated code, cmd/ per service, internal/platform/ for shared libs, web/ for Astro frontend
- Buf v1.66.0 for proto management (not raw protoc)
- pgx v5 for PostgreSQL (no ORM — raw SQL for sharded architecture control)
- franz-go for Kafka (not sarama — sarama is in maintenance mode)
- scylladb/gocql fork (not upstream gocql — shard-aware connection pooling)
- coder/websocket (not gorilla/websocket — gorilla is archived)

### Integration Points
- Envoy gRPC-JSON transcoder needs proto descriptor set generated by buf (deploy/envoy/proto.pb)
- Proto files must include google.api.http annotations for REST-to-gRPC mapping
- All services share internal/platform/ libraries (grpcserver bootstrap, config, database, redis, kafka, middleware, errors, pagination)
- Docker Compose orchestrates: PostgreSQL (initially 1 instance), Redis, Envoy, and the skeleton service
- Astro frontend lives in web/ with its own package.json. Communicates with Envoy gateway over REST/JSON

</code_context>

<specifics>
## Specific Ideas

- "Terminal-like UI, lyra style" — the entire frontend should feel like a TUI (text user interface) rendered in the browser. Think monospace fonts, box-drawing characters, information density, minimal decorative elements
- ASCII box borders should use CSS that mimics characters like +, -, |, or actual Unicode box-drawing characters (U+2500 series)
- The sidebar should feel like a file tree in a code editor — communities as navigable items, not flashy cards
- Dark mode should be the default experience (matching the terminal aesthetic), but light mode must work well too

</specifics>
