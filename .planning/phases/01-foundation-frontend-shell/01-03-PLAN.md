Create the shared internal/platform/ libraries that every service will import. Use the patterns from 01-RESEARCH.md.

**internal/platform/errors/errors.go** — Domain error types:
- Define sentinel errors using `errors.New()`: `ErrNotFound`, `ErrAlreadyExists`, `ErrForbidden`, `ErrInvalidInput`, `ErrUnauthenticated`
- These are the ONLY domain errors. Services return these; the interceptor maps them to gRPC codes.

**internal/platform/config/config.go** — Config loading:
- `type Config struct` with fields: `GRPCPort int`, `DatabaseURL string`, `RedisURL string`, `ServiceName string`
- `func Load(serviceName string) *Config` — reads from environment variables with sensible defaults
- Defaults: GRPCPort=50051, DatabaseURL=postgres://redyx:dev@localhost:5432/{serviceName}?sslmode=disable, RedisURL=redis://localhost:6379/0
- Use `os.Getenv()` directly — no external config library needed for Phase 1
- Log loaded config values (redact password from DatabaseURL) using zap

**internal/platform/database/postgres.go** — PostgreSQL helper:
- `func NewPostgres(ctx context.Context, databaseURL string) (*pgxpool.Pool, error)`
- Creates pgxpool with sensible defaults: MaxConns=25, MinConns=5, MaxConnLifetime=5min
- Pings the database to verify connectivity
- Returns the pool

**internal/platform/redis/client.go** — Redis helper:
- `func NewClient(redisURL string) (*redis.Client, error)`
- Parses the URL, creates client with timeouts: ReadTimeout=3s, WriteTimeout=3s, DialTimeout=5s
- Pings Redis to verify connectivity
- Returns the client

**internal/platform/middleware/logging.go** — Logging interceptor:
- gRPC UnaryServerInterceptor that logs: method name, duration, status code
- Uses zap structured logging (JSON format)
- Logs at Info for success, Warn for client errors (4xx equivalent), Error for server errors
- `func Logging(logger *zap.Logger) grpc.UnaryServerInterceptor`

**internal/platform/middleware/recovery.go** — Panic recovery:
- gRPC UnaryServerInterceptor that recovers from panics
- Logs the stack trace via zap
- Returns `codes.Internal` status to the client
- `func Recovery(logger *zap.Logger) grpc.UnaryServerInterceptor`

**internal/platform/middleware/errors.go** — Error mapping interceptor:
- gRPC UnaryServerInterceptor (exactly as in RESEARCH.md Pattern 2)
- Maps domain errors to gRPC status codes:
  - `ErrNotFound` → `codes.NotFound` (HTTP 404)
  - `ErrAlreadyExists` → `codes.AlreadyExists` (HTTP 409)
  - `ErrForbidden` → `codes.PermissionDenied` (HTTP 403)
  - `ErrInvalidInput` → `codes.InvalidArgument` (HTTP 400)
  - `ErrUnauthenticated` → `codes.Unauthenticated` (HTTP 401)
  - Any other error → `codes.Internal` with sanitized message "internal error" (never leak raw errors per Pitfall 11)
- If error is already a gRPC status, pass through unchanged
- `func ErrorMapping() grpc.UnaryServerInterceptor`

**internal/platform/grpcserver/server.go** — Server bootstrap:
- Follow RESEARCH.md Pattern 1 exactly
- `type Server struct` with grpc.Server, health.Server, port, logger
- `type Option func(*serverConfig)` for WithUnaryInterceptors, WithStreamInterceptors
- `func New(port int, logger *zap.Logger, opts ...Option) *Server`
  - Creates grpc.Server with ChainUnaryInterceptor
  - Registers health.Server and reflection.Server
  - Sets initial health status to SERVING
- `func (s *Server) Server() *grpc.Server` — access underlying server for service registration
- `func (s *Server) SetServingStatus(service string, status healthpb.HealthCheckResponse_ServingStatus)` — for setting per-service health
- `func (s *Server) Run() error` — listens on port, handles SIGTERM graceful shutdown
  - On SIGTERM: set health to NOT_SERVING, call GracefulStop()

**internal/platform/pagination/cursor.go** — Cursor pagination helpers:
- `func EncodeCursor(id string, createdAt time.Time) string` — base64-encode cursor data
- `func DecodeCursor(cursor string) (id string, createdAt time.Time, err error)` — decode cursor
- `func DefaultLimit(requested int32, defaultLimit int32, maxLimit int32) int32` — clamp limit to range

**internal/skeleton/server.go** — Skeleton service implementation:
- Implements the generated `HealthServiceServer` interface from gen/redyx/health/v1/
- `type Server struct` with db pool, redis client, logger
- `func NewServer(db *pgxpool.Pool, rdb *redis.Client, logger *zap.Logger) *Server`
- Implement `Check(ctx, req) (*HealthCheckResponse, error)`:
  - Ping PostgreSQL (`db.Ping(ctx)`)
  - Ping Redis (`rdb.Ping(ctx)`)
  - If both succeed: return SERVING status with version "0.1.0"
  - If either fails: return NOT_SERVING with error detail
  - Include server uptime in response

**cmd/skeleton/main.go** — Service entry point:
- Initialize zap logger (production config for JSON output)
- Load config via `config.Load("skeleton")`
- Connect to PostgreSQL via `database.NewPostgres()`
- Connect to Redis via `redis.NewClient()`
- Create grpcserver with all three middleware interceptors (Logging, Recovery, ErrorMapping)
- Register skeleton.Server as HealthServiceServer
- Call `srv.Run()` — blocks until shutdown

**deploy/docker/Dockerfile** — Multi-stage Go build:
```dockerfile
FROM golang:1.24-alpine AS builder
ARG SERVICE=skeleton
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /bin/service ./cmd/${SERVICE}

FROM alpine:3.21
RUN apk add --no-cache ca-certificates
COPY --from=builder /bin/service /bin/service
EXPOSE 50051
ENTRYPOINT ["/bin/service"]
```

Use Go 1.24 (latest stable) in Dockerfile. The SERVICE build arg allows this same Dockerfile to build any service.

- `match_incoming_request_route: true` (routes match REST paths, not gRPC paths — per Pitfall 2)
- `preserve_proto_field_names: false` (camelCase JSON output — per Pitfall 1)
- `always_print_primitive_fields: true` (don't omit zero values)
- `add_whitespace: true` (readable JSON)
- HTTP/2 on upstream cluster (per Pitfall 5):
  ```yaml
  typed_extension_protocol_options:
    envoy.extensions.upstreams.http.v3.HttpProtocolOptions:
      "@type": type.googleapis.com/envoy.extensions.upstreams.http.v3.HttpProtocolOptions
      explicit_http_config:
        http2_protocol_options: {}
  ```
- Listen on 0.0.0.0:8080 for API gateway
- Listen on 0.0.0.0:9901 for Envoy admin interface
- Route `/api/v1/` prefix to skeleton-service cluster
- Register `redyx.health.v1.HealthService` in transcoder services list
- Proto descriptor at `/etc/envoy/proto.pb`
- Add CORS headers for frontend development (allow origin *, methods GET/POST/PUT/DELETE/OPTIONS, headers content-type/authorization)

**docker-compose.yml** — Local development stack (Phase 1 minimal):
```yaml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: skeleton
      POSTGRES_USER: redyx
      POSTGRES_PASSWORD: dev
    ports: ["5432:5432"]
    volumes: [pg-data:/var/lib/postgresql/data]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U redyx -d skeleton"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
    command: redis-server --save 60 1
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5

  skeleton-service:
    build:
      context: .
      dockerfile: deploy/docker/Dockerfile
      args:
        SERVICE: skeleton
    environment:
      DATABASE_URL: postgres://redyx:dev@postgres:5432/skeleton?sslmode=disable
      REDIS_URL: redis://redis:6379/0
      GRPC_PORT: "50051"
    ports: ["50051:50051"]
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy

  envoy:
    image: envoyproxy/envoy:v1.37.0
    ports:
      - "8080:8080"
      - "9901:9901"
    volumes:
      - ./deploy/envoy/envoy.yaml:/etc/envoy/envoy.yaml:ro
      - ./deploy/envoy/proto.pb:/etc/envoy/proto.pb:ro
    depends_on:
      - skeleton-service

volumes:
  pg-data:
```

Key points:
- PostgreSQL and Redis have healthchecks — skeleton-service waits for them
- skeleton-service builds from the shared Dockerfile
- Envoy mounts the envoy.yaml config and proto.pb descriptor as read-only volumes
- Envoy depends_on skeleton-service (Envoy needs the upstream to be running)
