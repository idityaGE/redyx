<!-- Proto contract for search service -->
From proto/redyx/search/v1/search.proto:
- SearchPosts(SearchPostsRequest) → SearchPostsResponse
  - Request: query (string), community_name (optional), pagination
  - Response: results[] (SearchResult), pagination
  - SearchResult: post_id, title, snippet, author_username, community_name, vote_score, comment_count, created_at
- AutocompleteCommunities(AutocompleteCommunitiesRequest) → AutocompleteCommunitiesResponse
  - Request: query (min 2 chars), limit (default 10)
  - Response: suggestions[] (CommunitySuggestion: name, icon_url, member_count)

<!-- PostEvent from Plan 01 (consumed by indexer) -->

<task type="auto">
  <name>Task 1: Create Meilisearch client wrapper and Kafka post indexer</name>
  <files>internal/search/meili.go, internal/search/indexer.go</files>
  <action>
1. Create `internal/search/meili.go`:
   - `MeiliClient` struct wrapping `meilisearch.ServiceManager` (the main meilisearch-go client interface)
   - `NewMeiliClient(url, apiKey string) (*MeiliClient, error)` — creates client, configures "posts" index:
     - SearchableAttributes: ["title", "body"]
     - FilterableAttributes: ["communityName"]
     - SortableAttributes: ["createdAt", "voteScore"]
     - RankingRules: ["words", "typo", "proximity", "attribute", "sort", "exactness"]
   - `Search(ctx context.Context, query, communityName string, sort string, limit, offset int) ([]SearchResult, int64, error)`:
     - Builds meilisearch.SearchRequest with query, optional filter `communityName = 'value'`, sort, limit, offset
     - Uses AttributesToHighlight: ["title", "body"], HighlightPreTag/PostTag for snippet generation
     - Uses AttributesToCrop: ["body:80"] for snippet truncation
     - Returns parsed results and total hits count
   - `IndexPost(postID, title, body, authorUsername, communityName string, voteScore, commentCount int32, createdAt int64) error`:
     - Adds/updates document in "posts" index with these fields
   - `DeletePost(postID string) error`:
     - Deletes document from "posts" index
   - Internal SearchResult struct matching proto fields

2. Create `internal/search/indexer.go`:
   - `Indexer` struct with `*kgo.Client` and `*MeiliClient`
   - `NewIndexer(brokers []string, meili *MeiliClient, logger *zap.Logger) (*Indexer, error)`:
     - Consumer group: `"search-service.redyx.posts.v1"` (unique, follows existing pattern per [03-03] decision)
     - Topic: `"redyx.posts.v1"`
   - `Run(ctx context.Context) error`:
     - PollFetches loop (same pattern as vote consumer)
     - Deserialize PostEvent via proto.Unmarshal
     - Switch on event_type:
       - CREATED/UPDATED → meili.IndexPost(...)
       - DELETED → meili.DeletePost(...)
     - Log errors but continue processing (don't crash on single bad message)
   - `Close()` method
  </action>
  <verify>
    <automated>go build ./internal/search/...</automated>
  </verify>
  <done>MeiliClient configures index settings and provides Search/IndexPost/DeletePost. Indexer consumes PostEvents from Kafka and indexes to Meilisearch.</done>
</task>

<task type="auto">
  <name>Task 2: Create search-service gRPC server and entry point</name>
  <files>internal/search/server.go, cmd/search/main.go</files>
  <action>
1. Create `internal/search/server.go`:
   - `Server` struct with `meili *MeiliClient`, `redisClient` (for autocomplete cache), `logger *zap.Logger`
   - `NewServer(meili *MeiliClient, redisClient redis.Cmdable, logger *zap.Logger) *Server`
   - Implement `SearchPosts` RPC:
     - Extract query, community_name, pagination from request
     - Validate query is non-empty (return InvalidArgument if empty)
     - Determine sort from query params or default to relevance (Meilisearch default ranking)
     - Call meili.Search with parsed params
     - Map results to proto SearchResult messages
     - Return SearchPostsResponse with results + pagination
   - Implement `AutocompleteCommunities` RPC:
     - Validate query.length >= 2 (return InvalidArgument if less)
     - Use Redis ZRANGEBYLEX on sorted set key "communities:autocomplete" for prefix matching:
       - Range: `[{query}` to `[{query}\xff` with limit
     - If Redis cache miss or empty, fall back to Meilisearch search on a "communities" index
     - Return AutocompleteCommunitiesResponse with matched suggestions
   - Register as `searchv1.SearchServiceServer`
   - Mark both RPCs as public (no auth required) in the auth interceptor — anonymous users should be able to search. Use the same pattern as GetPost/ListPosts in post-service.

2. Create `cmd/search/main.go`:
   - Follow the exact pattern from `cmd/comment/main.go` (logger → config → dependencies → middleware → server → run)
   - Load config for "search" service
   - Create MeiliClient from config (MeilisearchURL, MeilisearchAPIKey)
   - Connect to Redis DB 7 (search-service's reserved DB per [05-RESEARCH] decision)
   - Create and register gRPC search server with middleware chain: Recovery → Logging → Auth → RateLimit → ErrorMapping
   - Start Kafka Indexer in a goroutine (same pattern as vote consumer in comment-service)
   - Port: 50058 (next available after comment-service on 50057)
   - On startup, seed the communities autocomplete cache: fetch all community names from community DB or Meilisearch community index. Alternatively, populate the Redis sorted set with community names on first AutocompleteCommunities call.
   - Populate community autocomplete: Create a "communities" Meilisearch index on startup. For v1, use community-service's database to seed — connect to community DB (read-only) and load community names + member_counts into both Redis sorted set and Meilisearch communities index. Use `CommunityDatabaseURL` config field already in config.go.
  </action>

  </verify>

</task>
