    1. **hasher.go** — Argon2id password hashing following research Pattern 2:
       - `Params` struct with Memory (64*1024 KiB), Iterations (1), Parallelism (4), SaltLength (16), KeyLength (32) per RFC 9106
       - `DefaultParams` package var
       - `HashPassword(password string, p *Params) (string, error)` — generates salt with crypto/rand, returns `$argon2id$v=19$m=65536,t=1,p=4$<salt>$<hash>` encoded string
       - `VerifyPassword(password, encodedHash string) (bool, error)` — parses encoded hash, recomputes, uses `crypto/subtle.ConstantTimeCompare`
       - `decodeHash(encodedHash string) (*Params, []byte, []byte, error)` — parses the $-delimited format

    2. **jwt.go** — JWT token management following research code example:
       - `JWTManager` struct with secret []byte and accessTTL time.Duration

<task type="auto">
  <name>Task 2: Implement AuthServiceServer and auth service entry point</name>
  <files>internal/auth/server.go, cmd/auth/main.go</files>
  <action>
    1. **server.go** — Implement all 7 RPCs of AuthServiceServer:
       - `Server` struct holding: `*pgxpool.Pool`, `*goredis.Client`, `*JWTManager`, `*OTPManager`, `*OAuthManager`, `EmailSender`, `*zap.Logger`
       - `NewServer(db *pgxpool.Pool, rdb *goredis.Client, jwtMgr *JWTManager, otpMgr *OTPManager, oauthMgr *OAuthManager, emailer EmailSender, logger *zap.Logger) *Server`
       - Embed `authv1.UnimplementedAuthServiceServer`

       **Register:** Validate email format + username (3-20 chars, alphanumeric+underscore) + password (8+ chars). Check username/email uniqueness in DB. Hash password with argon2id. INSERT into users table (auth_method='email', is_verified=false). Generate OTP, send via emailer. Create profile in user-service (for Phase 2: direct DB insert into user_profiles database since user-service may not be running yet — OR skip cross-service call and let user-service plan handle profile creation sync). Return user_id + requires_verification=true.
       IMPORTANT: For cross-service profile creation, use a simple approach: auth-service creates the auth user record only. The user-service will be responsible for ensuring profiles exist (create-on-first-access pattern). This avoids tight coupling.

       **Login:** Look up user by email. Verify password with argon2id. Check is_verified (if not, return ErrUnauthenticated with "account not verified" message). Generate JWT access token. Generate refresh token (UUID via google/uuid), hash with SHA-256, store in refresh_tokens table with 7-day expiry. Return access_token, refresh_token, expires_at, user_id.

       **VerifyOTP:** Call otpMgr.Verify. If valid, UPDATE users SET is_verified=true. Generate JWT + refresh token (same as login). Return verified=true + tokens.

       **RefreshToken:** Hash the provided refresh_token with SHA-256, look up in refresh_tokens table. If not found or expired, return ErrUnauthenticated. Look up user by user_id from token record. Delete old refresh token. Generate new JWT + new refresh token. Return new tokens.

       **Logout:** Hash the provided refresh_token, DELETE from refresh_tokens table. Return empty response. If token not found, still return success (idempotent).

       **ResetPassword:** Two-step flow based on which fields are set:
       - Step 1 (email only, token+new_password empty): Generate reset token (32 bytes crypto/rand, base64url encoded), store in Redis with key `reset:{email}` and 1hr TTL. Send via emailer. Return completed=false.
       - Step 2 (token+new_password set): Look up `reset:{email}` in Redis, verify token matches. Hash new password. UPDATE users SET password_hash. Delete Redis key. Return completed=true.

       **GoogleOAuth:** Call oauthMgr.Exchange. Look up user by google_id. If exists: generate tokens, return is_new_user=false. If not exists AND username provided: create user with auth_method='google', is_verified=true (Google verified the email), generate tokens, return is_new_user=false. If not exists AND no username: return is_new_user=true with empty tokens (client must call again with username).

       **Privacy (AUTH-08):** NONE of the response messages include email or auth_method fields. The proto already enforces this — verify the proto response types don't leak private data.

       - Register AuthServiceServer
       - Run server
