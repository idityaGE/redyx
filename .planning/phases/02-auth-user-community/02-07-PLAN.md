<interfaces>
<!-- API endpoints for auth -->
POST /api/v1/auth/register  → { email, username, password } → { userId, requiresVerification }
POST /api/v1/auth/login     → { email, password } → { accessToken, refreshToken, expiresAt, userId }
POST /api/v1/auth/verify-otp → { email, code } → { verified, accessToken, refreshToken, expiresAt }
POST /api/v1/auth/google    → { code, username? } → { accessToken, refreshToken, expiresAt, userId, isNewUser }
POST /api/v1/auth/reset-password → { email } → { completed: false }  (step 1)
POST /api/v1/auth/reset-password → { email, token, newPassword } → { completed: true }  (step 2)

    1. **register.astro** — Astro page shell:
       - Uses BaseLayout with title="Register"
       - Renders `<RegisterForm client:load />`

    2. **RegisterForm.svelte** — Terminal prompt style registration:
       - Uses AuthForm wrapper (or builds inline with same aesthetic)
       - Fields: email (text input), username (text input), password (password input)
       - Terminal label style: `email: ` `username: ` `password: ` (monospace, left-aligned, like CLI prompts)
       - Validation on submit: email format, username 3-20 chars alphanumeric+underscore, password 8+ chars
       - Inline error messages below each field: `> error: {message}` in red/orange (per locked decision)
       - On submit: call `api('/auth/register', { method: 'POST', body: JSON.stringify({ email, username, password }) })`
       - On success (requiresVerification=true): redirect to `/verify?email={email}` (pass email as query param so verify page knows which account)
       - On API error: display error message inline
       - Link to login page: `already have an account? /login`
       - Loading state: button shows `[registering...]`

    3. **login.astro** — Astro page shell:
       - Uses BaseLayout with title="Login"
       - Renders `<LoginForm client:load />`

    4. **LoginForm.svelte** — Terminal prompt style login:
       - Fields: email, password
       - On submit: call `login(email, password)` from auth store
       - On success: redirect to `/` (home)
       - On error: show inline error `> error: invalid credentials`
       - If error says "account not verified": show link to `/verify?email={email}`
       - Link to register: `new user? /register`
       - Link to reset: `forgot password? /reset-password`
       - Google OAuth button (if OAuth is configured): styled as terminal command `> login with google_` — links to Google OAuth consent URL (built client-side or via a redirect endpoint)
       - For Phase 2 Google OAuth flow: link opens `/api/v1/auth/google/consent` which the auth service could handle, OR build the Google OAuth URL client-side using a public client ID. Simpler: add a small endpoint or construct the URL client-side with GOOGLE_CLIENT_ID from a public env var.

    5. **verify.astro** — Astro page shell:
       - Uses BaseLayout with title="Verify"
       - Renders `<VerifyForm client:load />`

    6. **VerifyForm.svelte** — OTP verification:
       - Reads email from URL query param `?email=`
       - Single field: 6-digit code input (could be 6 individual boxes or one text field — Claude's discretion, simpler: one text field with maxlength=6)
       - On submit: call `api('/auth/verify-otp', { method: 'POST', body: JSON.stringify({ email, code }) })`
       - On success (verified=true, tokens present): call `loginWithTokens(accessToken, refreshToken, userId)`, redirect to `/`
       - On error: show `> error: invalid or expired code`
       - Show email being verified: `verifying: {email}`
       - "Didn't receive code?" hint text (can't resend in Phase 2 — just tell user to register again)
