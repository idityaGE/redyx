  <action>
    **1. Create post-service score consumer** at `internal/post/vote_consumer.go`:
    - `ScoreConsumer` struct with `shards *ShardRouter`, `rdb *redis.Client`, `logger *zap.Logger`, `kafkaClient *kgo.Client`
    - `NewScoreConsumer(brokers []string, shards *ShardRouter, rdb *redis.Client, logger *zap.Logger) (*ScoreConsumer, error)`
      - Creates franz-go consumer in group `post-service.redyx.votes.v1` for topic `redyx.votes.v1`
    - `Run(ctx context.Context) error` — starts consuming loop:
      - For each VoteEvent:
        1. If target_type != "post", skip (comments are Phase 4)
        2. Read authoritative counts from vote-service Redis (DB 5):
           - `SCARD votes:up:{target_id}` → upvotes
           - `SCARD votes:down:{target_id}` → downvotes
           - `newScore = upvotes - downvotes` (set-based, naturally idempotent per RESEARCH.md)
        3. Route to correct shard: determine community_id from event (or look up from DB if empty)
        4. If community_id is available: `shards.GetPool(community_id)` → direct shard query
        5. If community_id not available: query all shards for the post_id to find it
        6. Update post: `UPDATE posts SET vote_score = $newScore, upvotes = $upvotes, downvotes = $downvotes, hot_score = $hotScore WHERE id = $post_id`
           - hot_score = HotScore(newScore, post.created_at) — fetch created_at from the same row or use `UPDATE ... SET hot_score = 10000.0 * ln(GREATEST(1, 3 + $newScore)) / POWER(EXTRACT(EPOCH FROM (now() - created_at)) / 3600 + 2, 1.8)` computed in SQL
        7. Log processing stats

    **Important:** The score consumer connects to BOTH Redis DB 4 (post cache) AND Redis DB 5 (vote state reads). Create a second Redis client for vote-state reads.

    **2. Wire karma consumer into user-service** — modify `cmd/user/main.go`:
    - Import the `internal/vote` package for `KarmaConsumer`
    - After server setup but before `srv.Run()`:
      - Create a separate Redis client for karma deduplication (can use user-service's Redis DB 2, or a dedicated namespace)
      - Create `KarmaConsumer` with Kafka brokers, Redis client, user DB pool, logger
      - Start consumer in a goroutine: `go karmaConsumer.Run(ctx)`
    - On shutdown: cancel context to stop consumer
    - Add `KAFKA_BROKERS` env var support (already in config from Plan 01)

    **CRITICAL route ordering for ListPosts:**
    The ListPosts RPC is at `GET /api/v1/communities/{community_name}/posts`. This overlaps with the existing `/api/v1/communities` route that goes to community-service. Since Envoy transcoder uses `match_incoming_request_route: true`, and the community-service gRPC also handles `/api/v1/communities/*` routes, we need a specific route:
    ```yaml
    # Community posts feed — route to post-service (before community-service catch-all)
    - match: { path: "/api/v1/communities/", headers: [{name: ":method", exact_match: "GET"}] }
    ```
    Actually, with `match_incoming_request_route: true`, the transcoder matches the REST path to the proto HTTP annotation. Since both community-service and post-service have `/api/v1/communities/` prefixed routes, we need BOTH services registered in the transcoder AND the Envoy route for `/api/v1/communities` to go to a service that can handle both.

    **Simplest approach:** Route `/api/v1/communities` to community-service (as before). The ListPosts annotation `/api/v1/communities/{community_name}/posts` ends in `/posts` — add a regex route that matches `/api/v1/communities/.*/posts` to post-service, placed BEFORE the generic `/api/v1/communities` route.
    ```yaml
    - match:
        safe_regex:
          regex: "/api/v1/communities/[^/]+/posts.*"
      route:
        cluster: post-service
        timeout: 30s
    ```
