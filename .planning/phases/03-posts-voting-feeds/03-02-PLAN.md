  <name>Task 1: Redis vote store with Lua atomic operations</name>
  <files>internal/vote/redis.go</files>
  <action>
    Create `internal/vote/redis.go` implementing the Redis vote state management:

    **VoteStore struct:**
    ```go
    type VoteStore struct {
        rdb    *redis.Client
        logger *zap.Logger
    }
    ```

    **Lua script for atomic vote cast** — single script that handles all vote transitions:
    - Input KEYS: state key, up set key, down set key, score key
    - Input ARGV: user_id, new_direction ("up", "down", "none")
    - Logic:
      1. GET current state from state key
      2. If current == new_direction → return {0, current_score} (idempotent, VOTE-05)
      3. Remove user from old set (SREM), add to new set (SADD), update state key
      4. Compute delta: map transitions to deltas (nil→up=+1, up→down=-2, etc.)
      5. INCRBY score key by delta
      6. GET new score
      7. Return {delta, new_score, old_direction}
    - This MUST be a single Lua script for atomicity (VOTE-02 enforcement, prevents race conditions)

    **CastVote method:**
    ```go
    func (s *VoteStore) CastVote(ctx context.Context, userID, targetID, direction string) (delta int, newScore int, oldDirection string, err error)
    ```
    - Constructs Redis keys: `votes:state:{userID}:{targetID}`, `votes:up:{targetID}`, `votes:down:{targetID}`, `votes:score:{targetID}`
    - Runs Lua script
    - Returns delta (for Kafka event), newScore (for API response), oldDirection (for delta computation)

    **GetVoteState method:**
    ```go
    func (s *VoteStore) GetVoteState(ctx context.Context, userID, targetID string) (string, error)
    ```
    - GET `votes:state:{userID}:{targetID}` — returns "up", "down", or "" (none)

    **GetScore method:**
    ```go
    func (s *VoteStore) GetScore(ctx context.Context, targetID string) (int, error)
    ```
    - GET `votes:score:{targetID}` — returns current score (0 if not set)

    **BatchGetVoteStates method** (for feed responses — RESEARCH open question 3, option C):
    ```go
    func (s *VoteStore) BatchGetVoteStates(ctx context.Context, userID string, targetIDs []string) (map[string]string, error)
    ```
    - Pipeline GET for each `votes:state:{userID}:{targetID}`
    - Returns map[targetID]direction

    Use Redis DB 5 (per RESEARCH.md convention: auth=1, user=2, community=3, post=4, vote=5).

    **1. Kafka producer** at `internal/vote/kafka.go`:
    - Add `github.com/twmb/franz-go` and `github.com/twmb/franz-go/pkg/kadm` dependencies
    - `Producer` struct wrapping `*kgo.Client`
    - `NewProducer(brokers []string) (*Producer, error)` — creates franz-go client with async production
    - `PublishVoteEvent(ctx context.Context, event *commonv1.VoteEvent) error`:
      - Serialize VoteEvent to protobuf bytes
      - Produce to topic `redyx.votes.v1` with key = event.TargetId (ensures partition ordering per target)
      - Use `client.Produce()` (async, fire-and-forget) — don't block the vote API response on Kafka write. Log errors but don't fail the RPC.
    - `EnsureTopic(ctx context.Context) error`:
      - Use `kadm` to create topic `redyx.votes.v1` with 6 partitions, 1 replication factor, 7-day retention
      - Ignore "topic already exists" error
    - `Close()` to flush and close client

    **2. Vote gRPC server** at `internal/vote/server.go`:
    ```go
    type Server struct {
        votev1.UnimplementedVoteServiceServer
        store    *VoteStore
        producer *Producer
        logger   *zap.Logger
    }
    ```

    **Vote RPC (VOTE-01, VOTE-02, VOTE-03, VOTE-05)**:
    - Require auth (claims must be non-nil)
    - Validate: target_id non-empty, target_type is POST or COMMENT, direction is UP/DOWN/NONE
    - Map VoteDirection enum to string: UP→"up", DOWN→"down", NONE→"none"
    - Call `store.CastVote(ctx, claims.UserID, req.TargetId, direction)`
    - If delta == 0 (idempotent no-op, VOTE-05): still return current score
    - If delta != 0: publish VoteEvent to Kafka (async):
      ```go
      event := &commonv1.VoteEvent{
          EventId:     uuid.New().String(),
          UserId:      claims.UserID,
          TargetId:    req.TargetId,
          TargetType:  targetTypeStr, // "post" or "comment"
          AuthorId:    "", // TODO: look up from post data or pass from client — for v1, leave empty, karma consumer will look it up
          ScoreDelta:  int32(delta),
          CommunityId: "", // TODO: same as above
          OccurredAt:  timestamppb.Now(),
      }
      ```
      Note: AuthorId and CommunityId are needed by consumers. For the Vote RPC to be fast, we accept that these fields may be empty initially. The post-service consumer can look up the post to get author_id and community_id. Alternatively, the vote-service can cache post metadata from Redis. Use the pragmatic approach: the consumers handle lookup.
    - Return VoteResponse with new_score

    **GetVoteState RPC**:
    - Require auth
    - Call `store.GetVoteState(ctx, claims.UserID, req.TargetId)`
    - Map string to VoteDirection enum: "up"→UP, "down"→DOWN, ""→NONE (which is VOTE_DIRECTION_UNSPECIFIED or NONE)
    - Return GetVoteStateResponse

    **3. Karma consumer** at `internal/vote/consumer.go`:
    - `KarmaConsumer` struct — designed to be embedded in user-service
    - `NewKarmaConsumer(brokers []string, redisClient *redis.Client, db *pgxpool.Pool, logger *zap.Logger) *KarmaConsumer`
    - `Run(ctx context.Context) error` — starts franz-go consumer in group `user-service.redyx.votes.v1`
    - On each VoteEvent:
      1. If event.AuthorId is empty, skip (can't update karma without knowing author)
      2. Deduplication: `SADD karma:processed:{author_id} {event_id}` with 24h TTL on the set
      3. If SADD returns 1 (new event): `UPDATE users SET karma = karma + $delta WHERE id = $author_id`
      4. If SADD returns 0 (duplicate): skip
    - Handle consumer group rebalancing gracefully (franz-go handles this)
    - Log processing stats periodically
